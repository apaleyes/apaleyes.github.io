Emukit: Decision making under uncertainty

Andrei Paleyes

SciPy 2023

Hi everyone! My name is Andrei, very happy to be a part of SciPy 2023, at least virtually.
Today I will be presenting Emukit. Emukit is a Python library for methods of decision making under uncertainty. So before we talk about the package itself let's discuss what does "decision making under uncertainty" even mean.
Our life is surrounded with complex processes, ranging from training deep neural nets, doing an experiment in a chemical lab, to getting your daily workout in or even baking cookies. No doubt all of these are non-trivial processes that depend on many factors.
f(x) = y
Ultimately, we can say that each of these processes is a function, f(x)=y. It takes some input, e.g. hyperparameters of the model or amouts of ingridients in the recipe. And it produces an output that we can measure, for example accuracy of the model or number of cookies we sell in a day. The function is black-box, meaning we don't know exactly how it all works, even if we have some intuition.
f(x) = y
And we often want to do something with this process and with this function: learn it, optimise it, integrate it, etc. But the thing is, f(x) is very expensive - maybe due to actual money, or time involved, or wear-and-tear. So we just cannot afford to run this function on the all possible inputs. What to do?
One way to be smarter is to be Bayesian: 1. make initial assumption 2. Guess where a good point is 3. try this one input point 4. adjust your beliefs 5. go to 2 Repeating until we achieve our goal.
Revisiting "Decision making under unceratinty"

"Decision making"?

"Under unceratinty"?
So we did we say "decision making"? As f(x) is expensive, choosing where to try or evaluate the function next is an important decision. Why "under uncertainty"? First, noise-free observations of real life processes almost never exist. Secondly, our own beliefs aren't exact or precise. If we see f(x) in one point, it only gives us some idea of how f(x) may behave around this point. Hopefully this gives you some intuition about methods Emukit deals with.

Depending on what you'd like to do, there are specific flavors of this general process. If you'd like to find an optimum, you are doing Bayesian optimisation. If you'd like to learn the function better in some part of the input domain, that's experimental design or active learning. Finally, Bayesian quadrature is how you integrate f(x).

But the key thing is that all these methods, despite ahving different names, different goals and separate communities working on them, all conform to the same high level algorithm. And that's the realisation upon which Emukit is built.
We like to contextualise Emukit in the following way, The decision making process we just discussed can be seen as going from models to methods to tasks.

Models

  • Gaussian process
  • Bayesian neural network
  • Random forest
  • Multi-fidelity
Model is the belief we talked about. It is a data-driven representation of whatever data we have about the objective function, and our unceratinty about it. Usually a special type of models, known as Gaussian process, is used, but sometimes you can also see random forests, BNNs etc. Emukit is not a modeling framework, so we expect users to define the model in a framework of their choice, and plug it into Emukit.

Methods

  • Bayesian optimisation
  • Bayesian quadrature
  • Experimental design
Methods are the bread and butter of Emukit, the main focus of the library, so we'll talk about them more later.

Tasks

  • Hyperparameter optimisation
  • Data efficient learning
  • Benchmarking
Finally, there are tasks. These are meaningful things you are looking to do. Unless you are a student working on an assignment, you never really want to find an integral of a function or its maximum. You normally do that because you want to get better results from your experiments, reduce costs, etc. There is really no limit to applications of these decisoin making methods, so the library only provides tutorial and examples of things that could be done.
Diving deeper into Emukit, all the methods we mentioned consist of the same set of components. To begin, there is an objective function. As we said, there is a model, our internal representation of data we have on objective function. Then there is an acquisition function, which is a mechanism to use the model to decide where to evaluate the function next. It also balances exploration and exploitation of the process. Finally, there is a stopping criterion telling us when to stop. With Emukit we wanted to give our users a lot of flexibility as to what implementations of these components they use, make it easy to swap and compare. Basic idea was "it would be nice to just replace an acquisition function leaving everything else intact"

And we achieved this sort of flexibility by simply following good engineering practices. We use SOLID principles a lot, and some parts of code may remind you of certain gang of four patterns. For instance, we have a common abstract loop, where we can plug things in if they implement an interface required by the loop.

Similarly, if a component needs to interact with another component, they can also do that through interfaces. For example here an acquisition expects certain functionality from the model. The functionality is declared via an interface that has a couple of functions returning numpy arrays. This approach completley decouples acquistion functions and models. It also means the models can be developed based on any framework of choice. TensorFlow, PyTorch - no matter, as long as it implements the given interface.
Even from these two examples we can tell that there is no rocket science and crazy engineering trick here. Just applying basic principles allowed us to build an intuitive and flexible library.
Wait!

Numpy arrays...

Model framework agnostic...
But hold on a second, let's consider what I just said. Numpy arrays... Backend agnostic... From the biggest virtues of Emukit also comes its biggest drawback - performance!
The main packages we consider direct competition to Emukit are BoTorch and Trieste. Of course there are many more, but let's focus on these two as probably the strongest of the bunch. They are built on PyTorch and Tensorflow respectively, very tightly coupled with these backends. And thus enjoy all the features of these monsters, including GPU acceleration! Being backend agnostic Emukit simply doesn't have access to the GPU acceleration, as we cannot make any assumptions about how the underlying math is done. So if you require good speed, please take this into consideration.
But why use Emukit then?
But if performance is not Emukit's forte, then what is? Let's talk about it.
First of all, rapid prototyping. Emukit's original goal was to support researchers in developing new methods for decision making under uncertainty. Hence all the thought put into flexibility. Quickly trying things out is the intended purpose of Emukit, and we have had lots of good feedback from resaerchers about it.
Emukit also turned out to be pretty intuitive and expressive. Makes it great for teaching. We know that Emukit was and is used for multiple university courses and summer schools.
Amazingly, given the size of the scientific Python community and the speed at which it grows, there are some features that were unique to the package back when it was developed and still are now. Most notably, Emukit remains the only general purpose package for BQ and MF.

Future plans

  • Friendly to beginners
  • Feature completeness for UQ
  • Integrations
  • Ideas?
So this is where the package is now. What does the future hold? There are multiple possible directions. First, Emukit could be more friendly to absolute beginners. I'd love to be able to say "optimise your models with Emukit in 3 lines of code", but I simply cannot, because the package wasn't built that way. Second, there are functionalities, like sensitivity analysis and Bayesian calibration, that were always planned but never really implemented. These would make the package more complete as far as uncertainty quantification methods go. Third, integration with other tools. For example create custom optimisers with Emukit for Scipy, Ray Tune or Optuna. Finally, one of the reasons I am here (well, kind of here) today is to gather feedback from this amazing community. And maybe through some conversations and comments a new interesting direction for Emukit emerges.

What is dataflow architecture?

"Data flow schemas", Dennis et al., International Symposium on Theoretical Programming, 1974

Boring slide

Control flow: instructions are executed one after another, classic von Neumann architecture.

Dataflow: instruction is ready to execute as soon as all its inputs are available.

Intuition behind the boring slide

Control flow is about operations and their order

Dataflow is about data routes and transformations

Examples of dataflow in action

  • Flow-based programming
  • MapReduce
  • Data streaming

You are likely familiar with dataflow already!

Features of dataflow architecture

  • Separates data and logic
  • Data as a first class citizen
  • Complete dataflow graph
  • Decentralised
